# SpeedCalendar 项目协作注意事项

欢迎加入 SpeedCalendar 项目！为了我们能高效协作，减少代码合并时的冲突，请大家遵守以下约定：

## 1. 核心原则：按功能模块分工

我们已经将项目结构按功能模块进行了划分，每个人都在自己的专属“领地”上工作，可以最大程度地避免互相干扰。

*   **项目结构**:
    ```
    com.example.speedcalendar
    ├── features
    │   ├── home     // 首页模块
    │   ├── ai       // AI模块
    │   └── mine     // 我的模块
    └── ... (其他共享文件)
    ```

*   **明确分工**:
    *   **负责人A**：请在 `features/home` 包内进行开发。
    *   **负责人B**：请在 `features/ai` 包内进行开发。
    *   **负责人C**：请在 `features/mine` 包内进行开发。

**请尽量只修改和添加你所负责包内的文件。**

## 2. Git 工作流程：使用功能分支

请不要直接向 `main` (或 `master`) 分支推送代码。我们采用标准的 `Feature-Branch` 工作流。

1.  **开始新工作前，先同步主分支**:
    ```sh
    git checkout main
    git pull
    ```

2.  **为你的新功能创建一个专属分支**:
    分支命名可以清晰一些，比如 `feature/login-ui` 或 `fix/home-bug`。
    ```sh
    git checkout -b <你的功能分支名>
    ```

3.  **在你的分支上进行开发和提交**。
4.  **推送你的分支到 GitHub**。
5.  **创建 Pull Request (PR)**：在 GitHub 上，从你的功能分支向 `main` 分支发起一个 Pull Request，并指定其他成员进行审查（Review）。

这样做的好处是，所有的代码修改都会经过审查，能有效保证代码质量和项目稳定。

## 3. 需要特别注意的【共享文件】

有些文件不属于任何一个功能模块，而是整个项目共享的。**在修改这些文件前，请务必在群里告知大家你要做什么**，以避免冲突。

这些共享文件主要包括：

*   **Gradle 脚本**:
    *   `app/build.gradle.kts` (当需要添加新的依赖库时)
*   **导航文件**:
    *   `navigation/Screen.kt` (当需要定义新的页面路由时)
    *   `ui/MainScreen.kt` (主导航界面，通常不需要修改)
*   **公共资源文件**:
    *   `res/values/strings.xml`
    *   `res/values/colors.xml`
    *   `res/drawable/` 目录下的图标

## 4. 沟通是关键

*   遇到不确定的问题，或者需要修改共享文件时，**立即沟通**。
*   在开始一项比较大的修改前，最好先在群里打声招呼。

遵循以上约定，我们的协作将会非常顺畅。祝大家编码愉快！
